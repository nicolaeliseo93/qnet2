# Product Rules

## Purpose

Questo documento definisce le regole che guidano la progettazione e l'evoluzione del prodotto.

Ogni decisione relativa a nuove funzionalità deve rispettare questi principi.

---

# MVP First

Ogni nuova idea deve essere implementata nella forma più semplice possibile.

Prima versione:

- Piccola
- Funzionante
- Validabile

Versioni successive:

- Ottimizzazione
- Automazione
- Scalabilità

---

# Solve Real Problems

Ogni funzionalità deve risolvere un problema reale.

Prima di sviluppare una feature chiedersi:

- Chi la utilizzerà?
- Quale problema risolve?
- Quale beneficio genera?

Se non esiste una risposta chiara, la feature non deve essere sviluppata.

---

# Avoid Feature Creep

Non aggiungere funzionalità "perché potrebbero servire".

Implementare esclusivamente ciò che genera valore oggi.

Le funzionalità future devono rimanere nel backlog fino a quando non esiste una reale necessità.

---

# One Problem, One Solution

Una funzionalità deve avere un obiettivo preciso.

Evitare schermate o componenti che cercano di risolvere troppi problemi contemporaneamente.

---

# Validate Before Expanding

Prima di ampliare una funzionalità verificare:

- Utilizzo reale
- Feedback utenti
- Limiti attuali

Non investire tempo su funzionalità che non vengono utilizzate.

---

# User Experience Before Technology

Le decisioni devono essere guidate dall'esperienza dell'utente.

Non scegliere una soluzione solo perché tecnicamente interessante.

---

# Minimize User Effort

Ogni operazione deve richiedere il minor numero possibile di:

- Click
- Passaggi
- Campi da compilare
- Decisioni

---

# Consistency Over Creativity

Lo stesso comportamento deve essere mantenuto in tutto il sistema.

Esempi:

- Stessi pulsanti
- Stesse terminologie
- Stessi flussi operativi

---

# Data Integrity First

La correttezza dei dati è più importante della velocità di sviluppo.

Ogni funzionalità deve garantire:

- Dati coerenti
- Dati recuperabili
- Dati verificabili

---

# Backward Compatibility

Quando possibile evitare modifiche che interrompono funzionalità esistenti.

Valutare sempre l'impatto sui dati e sugli utenti.

---

# Build Incrementally

Preferire:

Versione 1
↓
Feedback
↓
Versione 2
↓
Feedback
↓
Versione 3

piuttosto che mesi di sviluppo prima del rilascio.

---

# Every Feature Has A Cost

Ogni funzionalità aggiunta aumenta:

- Complessità
- Costi di manutenzione
- Costi di test
- Costi di supporto

Una nuova funzionalità deve giustificare il proprio costo.

---

# Definition Of Done

Una funzionalità è completata solo quando:

- È sviluppata
- È testata
- È documentata
- È utilizzabile dall'utente
- È coerente con gli standard aziendali
